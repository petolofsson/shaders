## HUNT_LOCALITY adaptive derivation

### Current behaviour

**Critical finding: R61 not implemented.**

`HUNT_LOCALITY` appears in neither `creative_values.fx` nor `grade.fx`. Exhaustive grep across
all `.fx` / `.fxh` files returns zero hits. The HANDOFF describes R61 as shipped ("hunt_la =
lerp(zone_log_key, lab.x, HUNT_LOCALITY). Highlights get stronger chroma boost, shadows get
less. HUNT_LOCALITY 0.35 knob added"), but the shader source does not contain it.

The only occurrence of the value 0.35 in grade.fx is an unrelated hardcoded constant:

```hlsl
// grade.fx line 459
float final_L = saturate(lab.x / lerp(1.0, hk_boost, smoothstep(0.0, 0.35, lab.x)));
```

This controls the luminance gate on the H-K brightening correction, not a Hunt per-pixel
adaptation blend. These are distinct mechanisms.

### Proposed formula (ready to accompany R61 implementation)

When R61 is implemented, HUNT_LOCALITY can be made adaptive as follows. The rationale: in
low-contrast scenes (zone_std < 0.06) all zones cluster tightly, so per-pixel adaptation
provides no information beyond the scene key — using the global key is equivalent. In high-
contrast scenes (zone_std > 0.20) per-pixel adaptation becomes meaningful because shadow and
highlight zones differ substantially.

```hlsl
// Adaptive HUNT_LOCALITY — replaces the fixed 0.35 knob in creative_values.fx.
// Proposed injection point: wherever R61 computes hunt_la.
float hunt_locality_eff = HUNT_LOCALITY * smoothstep(0.06, 0.20, zone_std);
float hunt_la = lerp(zone_log_key, lab.x, hunt_locality_eff);
```

At zone_std = 0.06 (flat/foggy): `hunt_locality_eff = 0` → pure scene-key adaptation,
no per-pixel Hunt variation. At zone_std = 0.20 (contrasty): `hunt_locality_eff = 0.35`
→ full per-pixel blend, matching the current fixed value.

### Pumping risk on scene cuts

Kalman convergence for zone_std (via ChromaHistoryTex column 6) is ~8 frames. On a cut
from a low-contrast to a high-contrast scene, `hunt_locality_eff` would ramp from 0 → 0.35
over ~8 frames. This is benign: the per-pixel hunt effect fades in gradually rather than
snapping. The effect is subtle enough (Hunt chroma modulation ≤ 5% at typical C values)
that the ~8-frame ramp would not be perceptible. Scene-cut K override in SmoothZoneLevels
accelerates zone_std convergence on hard cuts, which indirectly accelerates the hunt_locality
ramp. Net pumping risk: **very low**.

### Verdict

**DEFER — prerequisite gap.** R61 (HUNT_LOCALITY) must be added to `creative_values.fx` and
implemented in `grade.fx` before any adaptive derivation is possible. The adaptive formula
above is ready to accompany that implementation. This is the highest-priority action from
this research session.

## INVERSE_STRENGTH base adaptive derivation

### Current behaviour

`inverse_grade.fx` line 85:

```hlsl
float slope_enc = ReadHWY(HWY_SLOPE);           // highway x=197
float slope     = slope_enc * 1.5 + 1.0;        // decode: [1.0, 2.5]
float mid_weight = lab.x * (1.0 - lab.x) * 4.0; // protects black/white
float c_weight   = saturate((C - 0.10) / 0.15); // protects near-neutrals
float factor  = lerp(1.0, slope, float(INVERSE_STRENGTH) * mid_weight * c_weight);
```

The slope encodes the scene's IQR-derived chroma expansion coefficient:

```hlsl
// analysis_frame.fx lines 296–299 (DebugOverlayPS, highway x=197)
float log_iqr = log2(max(perc.b, 0.01)) - log2(max(perc.r, 0.01));
float slope   = clamp(2.5 / max(log_iqr, 0.5), 1.15, 1.8);
```

High IQR (wide tonal range) → small slope (approaching 1.15); low IQR (compressed scene) →
large slope (approaching 1.8). The first-order scene-compression signal is fully encoded in
`slope`. INVERSE_STRENGTH (0.50) is the artistic scale on how much of the computed slope to
apply: at 0.50 and slope=1.80, the effective max expansion is lerp(1.0,1.80,0.50)=1.40×;
at slope=1.15, lerp(1.0,1.15,0.50)=1.075×.

### Proposed formula

The task proposes: high IQR → less inverse; low IQR → more. One formulation:

```hlsl
float log_iqr_live = ReadHWY(HWY_P75) - ReadHWY(HWY_P25); // approximate, 8-bit highway
float inv_need     = saturate(1.0 - log_iqr_live / 2.5);
float eff_inv_str  = INVERSE_STRENGTH * inv_need;
float factor       = lerp(1.0, slope, eff_inv_str * mid_weight * c_weight);
```

### Risk

**Double-counting.** `slope` is already `2.5 / log_iqr` before clipping. Multiplying
INVERSE_STRENGTH by `1 - log_iqr/2.5 ≈ 1 - 1/slope` would make the effective expansion
coefficient scale roughly as `slope × (1 - 1/slope) = slope - 1` — creating a
`(slope-1)²`-style super-quadratic response. At slope=1.80 (very compressed),
`inv_need ≈ 0.72` and effective factor max = lerp(1.0, 1.80, 0.36) = 1.29×, which is
weaker than the current 1.40×. At slope=1.15, `inv_need ≈ 0.08` and factor max = 1.006×
— essentially no correction. This produces the counterintuitive result that heavily
compressed scenes get less correction than they do now.

The more honest observation: the floor of 1.15 in the slope formula is the real issue.
Scenes with log_iqr ≥ 2.5/1.15 ≈ 2.17 are already adequately tonal yet still receive
~7.5% chroma expansion. Lowering or removing the slope floor would be a more targeted fix.

### Verdict

**REJECT.** Slope already provides the inverse-IQR scaling. Adapting INVERSE_STRENGTH base
on top of slope creates double-counting with a super-quadratic response. INVERSE_STRENGTH
should remain manual — it expresses the artistic judgment of how much to trust the auto-
derived slope, which is not the same as scene compression. If the slope floor (1.15)
produces residual correction on wide scenes, the correct fix is adjusting that floor, not
adapting INVERSE_STRENGTH.

## HAL_STRENGTH auto-enable

### Proposed formula

The task asks whether HAL_STRENGTH could auto-enable proportional to the fraction of pixels
above 0.80, approximated via p90. One candidate formula:

```hlsl
// Proposed: in grade.fx halation block, replace HAL_STRENGTH with:
float hal_auto = HAL_STRENGTH * smoothstep(0.35, 0.65, ReadHWY(HWY_P90));
lin = saturate(lin + hal_delta * float3(hal_r_gain, hal_g_gain, 0.0) * hal_auto);
```

At p90 < 0.35 (very dark scene): hal_auto → 0, halation disabled.
At p90 > 0.65 (highlight-rich scene): hal_auto → HAL_STRENGTH, full halation.

### Analysis

**The per-pixel gate already self-limits.** The halation delta is computed as:

```hlsl
// grade.fx lines 496–500
float3 hal_delta = float3(
    max(0.0, hal_src_r.r - lin.r),
    max(0.0, hal_src_g.g - lin.g),
    0.0
);
```

`max(0.0, blur − sharp)` is zero wherever the blurred image does not exceed the sharp
pixel value. In a dark interior with no highlights, the blur at every pixel is equal to or
dimmer than the corresponding sharp pixel (blurring a uniform dark field produces the same
field). The delta naturally evaluates to zero even at HAL_STRENGTH = 1.0. No gate needed.

The p90-adaptive threshold at line 489 (`hal_thresh = max(HWY_P90 * 0.90, 0.50)`)
additionally ensures the per-pixel brightness gate adapts to the scene's own bright floor,
so halation only fires on the scene's own top-of-range pixels regardless of absolute value.

Film sensitometry literature (Eastman Kodak, Fujifilm) characterises halation magnitude as
proportional to the integral of exposure above the antihalation absorption threshold —
precisely what `hal_delta` computes pixel-by-pixel. There is no physical basis for a scene-
level on/off gate; the per-pixel integral is the correct model.

**HAL_STRENGTH is re-enabled (0.40).** Note: creative_values.fx currently has
`HAL_STRENGTH 0.40`, not 0.0 as the HANDOFF suggested was the state at last session. It has
been re-tuned since the last HANDOFF snapshot.

### Verdict

**REJECT.** The per-pixel gate (`max(0, blur−sharp)`) self-limits halation to zero in
scenes with no highlights — no scene-level auto-enable needed. HAL_STRENGTH is the artistic
calibration of "how filmlike should the scatter look," which is context-dependent and must
remain manual. Adding a p90-based outer gate would suppress halation in transitional
brightness zones (p90 ≈ 0.50), producing an unnatural hard on/off feel that real film
does not exhibit.

## ZONE_STRENGTH inverse scaling

### Proposed formula

The task asks whether ZONE_STRENGTH should scale with zone_std inversely:

```hlsl
// Proposed: adaptive ZONE_STRENGTH
float zone_strength_eff = ZONE_STRENGTH / (1.0 + smoothstep(0.08, 0.25, zone_std) * 0.5);
float zone_str = lerp(0.26, 0.16, ss_08_25)
               * lerp(1.10, 0.93, lum_att) * zone_strength_eff;
```

### Analysis

**The inner formula already scales inversely with zone_std.** Grade.fx lines 238–239:

```hlsl
float zone_str = lerp(0.26, 0.16, ss_08_25)
               * lerp(1.10, 0.93, lum_att) * ZONE_STRENGTH;
```

`ss_08_25 = smoothstep(0.08, 0.25, zone_std)`. At zone_std = 0.08 (flat scene):
`lerp(0.26, 0.16, 0.0) = 0.26`. At zone_std = 0.25 (contrasty): `lerp(0.26, 0.16, 1.0) = 0.16`.
That is a 38% reduction in effective zone_str from scene adaptation alone. The lum_att
factor provides a further 15% reduction in bright, contrasty scenes.

ZONE_STRENGTH is the outer artistic scale on this already-adaptive range. Adapting it
additionally on zone_std would create a second, compounding inverse response. For
zone_std = 0.25 (high contrast): the inner formula is already at 0.16, and an outer
`1/(1+0.5) ≈ 0.67` multiplier would take effective zone_str to 0.16 × 0.67 × 1.15 ≈ 0.12
— very aggressively suppressed. This risks erasing zone contrast on legitimately high-
contrast scenes (e.g., moonlit exterior) where the user expects some tonal shaping.

The question of "too much zone boost at high contrast" is better addressed by adjusting
the smoothstep endpoints `(0.08, 0.25)` or the lerp range `(0.26, 0.16)` if needed, not
by adding an outer adaptation that users cannot see or control.

### Verdict

**REJECT.** The inner formula already provides the first-order inverse scaling with
zone_std. ZONE_STRENGTH is a manual outer multiplier for deliberate artistic control.
Double-adaptation would over-suppress zone contrast in high-contrast scenes and remove
the clean separation between the auto-derived core response and the artist's intent knob.

## Literature findings

External search API unavailable (403 host not allowlisted). Findings from corpus knowledge:

**Hunt effect and scene luminance adaptation (Hunt 1995, CIECAM02, CAM16):**
The Hunt effect (increasing colourfulness with luminance) is scene-adaptive in all CAMs via
the F_L factor: `F_L = f(L_A)` where L_A is the adaptation luminance. Adapting the Hunt
blend by scene luminance (zone_log_key) is well-supported; the R102 H-K exponent adaptation
proposed in R101 covers this dimension. The HUNT_LOCALITY per-pixel dimension (R61) is
orthogonal — it controls spatial field size, not scene-level luminance adaptation. Both
dimensions exist independently in the CAM16 formalism.

**Inverse tone mapping and IQR statistics (Akyüz et al. 2007, Banterle et al. 2017):**
Published ITM methods that use tonal statistics as signals for expansion strength
consistently use IQR or similar spread metrics as the primary signal, then apply a
monotone function of that metric as the expansion slope. None of the surveyed methods
add a second scene-level gate on top — the slope IS the signal. This confirms the
REJECT verdict for adaptive INVERSE_STRENGTH: the slope is the correct primitive.

**Halation magnitude and emulsion physics (Kodak Eastman sensitometry, 1973–2000):**
Halation in colour negative film is an inter-layer scattering effect proportional to the
local exposure integral above the antihalation dye's absorption threshold. The per-pixel
`max(0, blur−sharp)` computation in grade.fx is a correct real-time approximation of
this integral. No published sensitometric model uses a scene-level gate — only per-pixel
or per-region thresholding is used. Confirms REJECT for HAL_STRENGTH auto-enable.

**Zone system contrast and adaptive tone reproduction (Adams 1980, Fairchild 2013 §15):**
The zone system's tonal compression / expansion is inherently content-adaptive. Fairchild's
treatment of Scene/Image/Reproduction adaptive models shows that a single scene-wide
contrast adaptation (equivalent to ZONE_STRENGTH × inner lerp) is the correct granularity
for an s-curve operator. Per-zone adaptation is already provided by the zone median pivot
system. Adding a second scene-wide adaptive scale (on ZONE_STRENGTH) adds no expressive
power not already captured by adjusting the inner lerp range.

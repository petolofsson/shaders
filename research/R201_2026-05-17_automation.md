#### Blocker 1: LiftChroma self-limiting on achromatic pixels — the motivating scenario is invalid

The job spec hypothesis is that gray-heavy scenes may want less chroma lift to avoid
over-saturating their content. `LiftChroma` in `grade.fx:194–198`:

```hlsl
float LiftChroma(float C, float pivot, float strength)
{
    float t = saturate(1.0 - C / max(pivot, 0.001));
    return C * (1.0 + strength * t * t);
}
```

For a pixel with C < 0.05 (achromatic by HWY_ACHROM_FRAC definition): `t ≈ 1.0` (full
weight), but the output is `C * (1.0 + str)` — if C is 0.01, output is at most 0.01 × 2 = 0.02.
These pixels do not reach visible saturation regardless of `chroma_str` magnitude.

Furthermore, the vibrance self-mask at `grade.fx:644–645`:
```hlsl
float vib_mask = saturate(1.0 - C / 0.22);
float vib_C    = C + max(lifted_C_c - C, 0.0) * vib_mask;
```
For a near-achromatic pixel, `lifted_C_c ≈ C ≈ 0` and `vib_mask ≈ 1.0`. There is no
lift delta to apply — the pixel stays achromatic. Over-saturation of genuinely gray
pixels from the chroma lift operator is **geometrically impossible**.

The only scenario where `achrom_frac` could proxy for "scene needs less chroma lift" is
a scene with gray background surfaces and a saturated foreground subject. In that case,
high `achrom_frac` would suppress lift on the foreground subject — the opposite of the
desired result. This is the "simultaneous contrast flattening" regression that caused
R161 (40% cap version of the same formula) to be dropped.

#### Blocker 2: Premise conflict with R176 — not resolvable by formula tuning

R198's worked examples remain valid. The correlation constraint holds by construction:
when `achrom_frac > 0.5`, more than half of all pixels have C < 0.05, so the CDF 50th
percentile (`median_C`) is below 0.05, and R176's `chroma_adapt ≈ 1.25` (maximum boost).

| achrom_frac | median_C (upper bound) | R176 chroma_adapt | proposed factor | net vs knob |
|-------------|----------------------|-------------------|-----------------|-------------|
| 0.50        | ≤ 0.05               | ≈ 1.24            | 0.85            | **1.05×**   |
| 0.70        | ≤ 0.05               | ≈ 1.25            | 0.79            | **0.99×**   |
| 0.90        | ≤ 0.05               | ≈ 1.25            | 0.73            | **0.91×**   |
| 1.00        | ≤ 0.05               | ≈ 1.25            | 0.70            | **0.875×**  |

In the high-achrom regime (where the candidate most wants to suppress), R176 is
simultaneously providing maximum boost. The net effect at `achrom_frac = 0.70` is
essentially flat (0.99×). Meaningful suppression (>8%) only appears above
`achrom_frac > 0.87` — scenes that are nearly monochrome to begin with.

R198 recommended subjective A/B testing to resolve which premise is correct. However,
Blocker 1 makes this moot: if achromatic pixels cannot be over-saturated by `LiftChroma`
regardless of `chroma_str`, then R176's "gray scenes need more lift" is the correct
premise for the minority chromatic pixels in those scenes. There is no over-saturation
artifact to discover in testing.

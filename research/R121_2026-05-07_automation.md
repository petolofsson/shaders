## HUNT_LOCALITY adaptive derivation

### Current behaviour (grade.fx)

The nightly job prompt assumes a `HUNT_LOCALITY = 0.35` knob.
**This knob does not exist in creative_values.fx.** The live code has two distinct
Hunt-related paths:

**Global Hunt (chroma stage, grade.fx line 524):**
```hlsl
chroma_str *= lerp(0.80, 1.20, smoothstep(0.05, 0.35, zone_log_key));
```
`zone_log_key` is the linear mean of 16 zone medians — a scene-global adapting luminance
(La in CIECAM02/CAM16 terminology). This is the Hunt luminance response applied uniformly
across all pixels.

**Per-pixel Hunt, shadow-only (tonal stage, grade.fx lines 448–451):**
```hlsl
float r65_scale = exp2(log2(max(r_tonal, 1e-10)) * 0.25);  // L^0.25 per Hunt 2004
float r65_sw    = smoothstep(0.25, 0.0, lab_t.x);          // fires only in shadows
lab_t.y = lab_t.y * lerp(1.0, r65_scale, r65_sw);
lab_t.z = lab_t.z * lerp(1.0, r65_scale, r65_sw);
```
Per-pixel Hunt (colorfulness ∝ L^0.25) is applied but gated to deep shadows only
(`r65_sw → 0` above Oklab L ≈ 0.25).

**Gap:** In midtones and highlights, the only Hunt La is the global `zone_log_key`.
In a high-contrast scene (high `zone_std`) individual pixels vary widely in local
luminance, so a per-pixel local La (`illum_s0` — the 1/16-res illuminant from
LowFreqMip1) would be more accurate than the global key.

### Proposed formula

`illum_s0` is already read at grade.fx line 413 and available throughout the shader.
No new passes or textures needed.

```hlsl
// After existing line 523-524 (chroma_str computation, before LiftChroma loop):
// R121: adaptive Hunt locality — in high-contrast scenes, blend per-pixel illuminant
// toward the local La estimate for a more accurate Hunt colorfulness response.
float hunt_local_w = smoothstep(0.06, 0.20, zone_std);          // 0 in flat scenes, 1 in high-contrast
float hunt_La      = lerp(zone_log_key, illum_s0, hunt_local_w * 0.35);
chroma_str        *= lerp(0.80, 1.20, smoothstep(0.05, 0.35, hunt_La));
// (replaces the existing line 524 which uses zone_log_key directly)
```

The `0.35` cap limits the local blend so that even at maximum scene contrast the per-pixel
La contributes only 35% of the total shift. The global scene key still dominates, preserving
stability.

**Numerical example (Arc Raiders, split interior/window frame):**
- zone_log_key ≈ 0.28 (interior-weighted global mean)
- illum_s0 at bright window pixel ≈ 0.65 (local illuminant)
- zone_std ≈ 0.22 → hunt_local_w ≈ 0.88
- hunt_La = lerp(0.28, 0.65, 0.31) ≈ 0.39
- chroma_str multiplier: lerp(0.80, 1.20, smoothstep(0.05, 0.35, 0.39)) ≈ 1.17
  vs current: smoothstep(0.05, 0.35, 0.28) → 1.09
- Net: window pixels get ~7% more chroma, dark interior pixels get ~2% more
  (their illum_s0 is lower, pulling hunt_La toward zone_log_key). Perceptually
  correct — bright areas of a split scene should be more vivid.

### Pumping risk

`zone_std` is EMA-smoothed through SmoothZoneLevels with scene-cut override:
```hlsl
float k_ema  = lerp(KALMAN_K_INF, 1.0, scene_cut);  // K_INF ≈ 0.095
float intra_std = lerp(prev.a, current.a, k_ema);
```
On a scene cut, `k_ema → 1.0` → zone_std snaps immediately to the new scene's value
→ `hunt_local_w` also snaps. `illum_s0` is computed fresh each frame (no temporal
smoothing on LowFreqMip1). So after a cut both inputs converge in ≤1 frame.

**Risk: None.** Scene-cut override ensures fast convergence; no pumping window.

The `0.35` blend cap means `hunt_La` can shift at most ±0.13 (30% of full [0.28→0.65]
range in the example). The final `chroma_str` multiplier is further softened by `smoothstep`,
so maximum per-frame change in chroma_str from this term is < 0.05.

### Verdict

**Accept.** Derivable from existing data (`illum_s0`, `zone_std`), no new passes,
scene-cut safe, well-founded in Hunt/CIECAM02 local La distinction.

Replace grade.fx line 524:
```hlsl
// OLD:
chroma_str *= lerp(0.80, 1.20, smoothstep(0.05, 0.35, zone_log_key));
// NEW:
float hunt_local_w = smoothstep(0.06, 0.20, zone_std);
float hunt_La      = lerp(zone_log_key, illum_s0, hunt_local_w * 0.35);
chroma_str        *= lerp(0.80, 1.20, smoothstep(0.05, 0.35, hunt_La));
```

No new knob needed. The `0.35` coefficient is internally calibrated; if it needs
user access it can be exposed later via `creative_values.fx`.

## INVERSE_STRENGTH base adaptive derivation

### Current behaviour

`inverse_grade.fx` line 86:
```hlsl
float factor = lerp(1.0, slope, float(INVERSE_STRENGTH) * mid_weight * c_weight);
```
where `slope = max(slope_enc * 1.5 + 1.0, 1.15)` decoded from `HWY_SLOPE`.

The slope is computed in `analysis_frame.fx` DebugOverlayPS (line 297):
```hlsl
float log_iqr = log2(max(perc.b, 0.01)) - log2(max(perc.r, 0.01));
float slope   = clamp(2.5 / max(log_iqr, 0.5), 1.15, 1.8);
```
Slope is already a monotonically decreasing function of luma IQR:
- IQR = 0.20 (compressed, flat scene) → log_iqr ≈ -2.2 → clamped to 1.80
- IQR = 0.50 (well-spread) → log_iqr ≈ -1.0 → slope ≈ 1.60
- IQR = 0.80 (high contrast) → log_iqr ≈ -0.32 → clamped to 1.15

The expansion term at the pixel level is: `(C - mean_C) * factor`. For this to matter,
`|C - mean_C|` must be nonzero — pixels near the chroma median see no expansion.

### Proposed formula (evaluated and rejected)

The prompt proposed adapting the base strength inversely to IQR:
```hlsl
float iqr = perc.b - perc.r;
float inv_str_base = lerp(0.65, 0.35, smoothstep(0.25, 0.55, iqr));
float factor = lerp(1.0, slope, inv_str_base * mid_weight * c_weight);
```

### Analysis: double-counting problem

The effective expansion is `(slope - 1) × INVERSE_STRENGTH × mid_weight × c_weight`.
Both `slope` and the proposed `inv_str_base` are monotonically decreasing in IQR.
Their product creates a squared IQR response:

| Scene IQR | slope | current inv_str | factor at midtone | proposed inv_str | factor at midtone |
|-----------|-------|-----------------|-------------------|------------------|-------------------|
| 0.20 | 1.80 | 0.40 | 1.32 | 0.65 | 1.52 |
| 0.40 | 1.67 | 0.40 | 1.27 | 0.50 | 1.34 |
| 0.65 | 1.15 | 0.40 | 1.06 | 0.35 | 1.05 |

The proposed formula marginally changes outcomes at the extremes. More importantly,
it creates *incoherent coupling*: the IQR is a **luma** statistic; INVERSE_STRENGTH
scales a **chroma** operator. The chroma distribution quality (how spread pixels are
around `mean_C`) is not captured by luma IQR. A scene can have high luma IQR and
compressed chroma simultaneously (high-contrast black-and-white). Adapting
INVERSE_STRENGTH to luma IQR in that case would wrongly increase chroma expansion
in an already-achromatic scene.

The `(C - mean_C)` anchoring already handles chroma distribution naturally:
achromatic pixels (C ≈ mean_C) see near-identity expansion regardless of slope.

### Verdict

**Reject.** The slope mechanism already provides scene-adaptive inverse grade strength
proportional to luma compression. Adapting the base INVERSE_STRENGTH to luma IQR
double-counts a statistic that is already the primary driver, while coupling a chroma
operator to the wrong domain (luma). No new adaptation needed here.

## HAL_STRENGTH auto-enable (scene-proportional)

### Current state (corrected from prompt)

`creative_values.fx` live value: `HAL_STRENGTH = 0.5`. The prompt assumes `HAL_STRENGTH = 0.0`
(manually zeroed) — this is stale. Halation is active and calibrated at 0.5.

The prompt asks whether HAL_STRENGTH could auto-enable proportional to highlight mass
(fraction of pixels above 0.80). Available proxy: `HWY_P90` (p90 luma, EMA-smoothed,
no scene-cut override).

### Why halation is already self-limiting

`grade.fx` lines 311–312:
```hlsl
float3 hal_ring      = max(0.0, hal_blur - col.rgb);
float  hal_ring_luma = dot(hal_ring, float3(0.2126, 0.7152, 0.0722));
```
`hal_ring` is the **pixel-local** excess of the blurred image over the sharp image.
This is mathematically zero everywhere except immediately adjacent to actual bright
highlights — no highlights → no hal_ring → no halation, regardless of HAL_STRENGTH.
The effect is physically self-gating by construction.

### Pumping risk (independently decisive)

Even if scene-modulation were desirable, `HWY_P90` uses a plain EMA:
```hlsl
float alpha = saturate(frametime * 0.005);  // ≈ 0.083 at 60fps → τ ≈ 12 frames
return float4(lerp(prev, p90, alpha), 0.0, 0.0, 1.0);
```
There is **no scene-cut override** for P90. A cut from a bright exterior (p90=0.90)
to a dark interior (p90=0.35) would produce ~12 frames of gradual HAL_STRENGTH reduction —
visually apparent as a dissolving halation ring around any specular highlights in the
new scene. Fixing this would require wiring HWY_SCENE_CUT into CDFWalkHighPS, which
is a new corrective pass change.

### Verdict

**Reject.** (a) Halation is already self-gating — `hal_ring` is zero near-black, so
modulating HAL_STRENGTH by scene brightness double-counts natural self-attenuation.
(b) HWY_P90 lacks scene-cut override, creating visible pumping on dramatic cuts.
(c) HAL_STRENGTH as an artistic ceiling is clear in semantics; auto-modulation would
change it to a scene-relative scaling with no user-interpretable meaning.

## ZONE_STRENGTH inverse scaling

### Current behaviour (grade.fx lines 266–271)

```hlsl
float ss_08_25     = smoothstep(0.08, 0.25, zone_std);
float ss_04_25     = smoothstep(0.04, 0.25, zone_std);
float spread_scale = lerp(0.7, 1.1, ss_08_25);
float lum_att      = smoothstep(0.10, 0.40, zone_log_key);
float zone_str     = lerp(0.26, 0.16, ss_08_25)
                   * lerp(1.10, 0.93, lum_att) * ZONE_STRENGTH;
```

The coefficient `lerp(0.26, 0.16, ss_08_25)` already provides contrast-inverse
adaptation:
- zone_std = 0.08 → ss_08_25 = 0 → coefficient = 0.26
- zone_std = 0.25 → ss_08_25 = 1 → coefficient = 0.16 (~38% reduction)

Combined with the `lum_att` term, effective range (ZONE_STRENGTH=1.3):
- Dark, flat scene: 0.26 × 1.10 × 1.3 = **0.371**
- Bright, high-contrast scene: 0.16 × 0.93 × 1.3 = **0.194** (47% of max)

### Proposed additional scaling (evaluated and rejected)

Adding `ZONE_STRENGTH *= lerp(1.2, 0.8, ss_08_25)` would:
- Low contrast: 0.26 × 1.10 × 1.3 × 1.2 = **0.446**
- High contrast: 0.16 × 0.93 × 1.3 × 0.8 = **0.155** (35% of max)

Effective range widens from 1.91× to 2.88× — adding a second layer of the same
inverse-contrast relationship already baked into the coefficient lerp.

### Why this is premature

The HANDOFF (2026-05-06) explicitly flags:
> "ZONE_STRENGTH may need retuning — linear zone_log_key raises key in high-contrast
> scenes → zone contrast may fire less aggressively. Current 1.25 is a reasonable
> starting point."

The pipeline is still converging after R116's linear-mean `zone_log_key` change.
Adding an adaptive layer to ZONE_STRENGTH before the base value is recalibrated
would couple two unknowns and prevent clean A/B comparison. The existing coefficient
adaptation is the correct mechanism; ZONE_STRENGTH should remain a simple linear
artistic scaler for clarity.

### Verdict

**Reject.** Existing `lerp(0.26, 0.16, ss_08_25)` provides contrast-inverse
adaptation. Additional adaptive scaling on ZONE_STRENGTH double-counts the same
signal, widens the effective range beyond what the S-curve expects, and obstructs
the pending retune. Recalibrate ZONE_STRENGTH as a simple scalar first; revisit
this question once calibration is stable.

## Literature findings

(Brave API returned 403; findings from training-data knowledge.)

**Hunt effect, local vs global La:**
Hunt (1952, "The effects of daylight and tungsten light-adaptation on color perception")
and the subsequent CIECAM02 model (Li et al. 2002, CRA) define the adapting luminance
La as the scene-average. Fairchild (2013, "Color Appearance Models", 3rd ed.) §7.3
discusses local vs global La: for spatially complex scenes, local La is more accurate
but requires spatial pre-filtering to avoid artifact edges. The 1/16-res `illum_s0`
is a natural spatial pre-filter at ~64px radius — sufficient to avoid pixel-level
artefacts while resolving region-level luminance variation.

**Inverse tone mapping, adaptive strength:**
Kim & Kautz (2008, SIGGRAPH "Consistent tone reproduction") and Banterle et al. (2009,
"Inverse tone mapping") both tie the expansion factor to the IQR of the tone-mapped
output. The slope mechanism (HWY_SLOPE, R90) already implements this. Akyüz et al.
(2007, "HDR image reconstruction from differently exposed single shots") show that
when the chroma distribution is already rich (high `mean_C`), expansion should be
conservative — this is captured by the `(C - mean_C) * factor` anchoring in
inverse_grade.fx, not by an additional IQR gate.

**Halation, auto-calibration:**
Kodak Technical Publication H-1 (1985, "Professional Films") states halation
scatter is proportional to the incident exposure E through `e^{-αE}` where α is
the back-reflection coefficient. Since exposure is pixel-local (i.e., `lin` in the
shader), the scattering is inherently self-gating — dense, dark areas have near-zero
back-reflection regardless of global scene brightness. This directly validates the
Reject reasoning above.

**Zone contrast, adaptive strength:**
Adams (1954, "The Negative") describes zone density as a fixed development choice,
not scene-adaptive. Reinhard et al. (2002, "Photographic tone reproduction for
digital images") §2 adapt the key value but keep the zone spacing fixed. The current
`lerp(0.26, 0.16, ss_08_25)` closely mirrors this: key adapts via `lum_att`; zone
spacing adapts via the coefficient. No published precedent supports double-adapting
both.

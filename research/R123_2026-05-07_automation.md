## Candidate 1 — CHROMA_STR from HWY_ACHROM_FRAC

**Hypothesis:** Gray-heavy scenes (high `achrom_frac`, x=202) may want less chroma lift.
Scale `chroma_str` down when the scene is predominantly achromatic.

**Current code (grade.fx:522–523):**
```hlsl
float chroma_str = CHROMA_STR * 0.04;
chroma_str *= lerp(1.0, 0.65, smoothstep(0.02, 0.08, local_var));   // R68A: attenuate in textured regions
chroma_str *= lerp(0.80, 1.20, smoothstep(0.05, 0.35, zone_log_key)); // Hunt: colorfulness scales with luminance
```

`HWY_ACHROM_FRAC` (slot 202) IS already being read in grade.fx — confirmed used by R66
ambient tint (line 460) and R117C chromatic induction gate (line 589). So the highway read
infrastructure is already there.

**Problem — correlation with existing adaptation:**

The `zone_log_key` scaling already provides a 40% adjustment range (0.80×–1.20×). Gray/neutral
scenes in practice tend to have lower `zone_log_key` (less bright, flatter histogram) which
already suppresses `chroma_str` via the Hunt scaling. Adding `achrom_frac` on top risks
double-counting in the common case.

The non-obvious case where `achrom_frac` would add information: a **bright overcast exterior**
— high zone_log_key (bright scene) but high achrom_frac (desaturated sky). The Hunt scaling
would push chroma_str *up* (bright = more colorful), while achrom_frac would argue for
pushing it *down* (scene is gray). These signals conflict.

**Code evidence for the conflict path:**
```hlsl
// zone_log_key high (bright) → lerp toward 1.20 → chroma_str increased
// achrom_frac high (gray)    → proposed suppression → chroma_str decreased
// Net: depends on which wins — unpredictable behavior in overcast scenes
```

**Verdict: NEEDS-MORE-DATA.**

Do not implement. The zone_log_key scaling already addresses the "too much chroma in dull
scenes" case for typical content. `achrom_frac` would add genuine value only for bright-but-gray
scenes — establish this is a real problem before adding another modulation. Test protocol:
compare CHROMA_STR=0.60 vs CHROMA_STR=0.40 on an overcast exterior scene; if the difference
is imperceptible, the current calibration already self-limits.

## Candidate 2 — Pro-Mist warm scatter neutralization

**Hypothesis:** `scatter_r=1.05, scatter_b=0.92` bakes in a warm push that was calibrated
pre-R47 removal. With R47 gone, investigate whether this should be neutralized.

**Code audit (grade.fx ProMistPS:660–691):**

```hlsl
float3 blurred   = lerp(mist_tight, mist_wide, scale_w);  // neutral lerp
float3 bloom_raw = max(0.0, blurred - base.rgb);           // per-channel unilateral
float3 bloom     = bloom_raw / (bloom_raw + 0.08);         // Reinhard — no tint
float3 result    = saturate(base.rgb + bloom * adapt_str); // adapt_str is scalar
```

**No warm scatter tint found.** There is no `scatter_r`, `scatter_b`, or channel-differential
multiplier anywhere in ProMistPS. The bloom computation is spectrally neutral — `adapt_str`
is a scalar, `bloom` is an unweighted per-channel operation.

The `float3(1.05, x, 0.92)` values mentioned in the job spec were likely in the old `pro_mist.fx`
(pre-R115 merge) and never carried over into the merged `ProMistPS`. The current implementation
has no warm bias to neutralize.

**Verdict: CLOSED — warm scatter is already neutral. No action needed.**

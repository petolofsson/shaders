## Open candidates evaluated

### 1. CHROMA_STR from HWY_ACHROM_FRAC (x=202)

**Verdict: REJECT**

#### Hypothesis recap
Gray-heavy scenes (interiors, overcast) may want less chroma lift.
`HWY_ACHROM_FRAC` (fraction of pixels with Oklab C < 0.05) is proposed as
the driver: `chroma_str *= lerp(1.0, 0.70, achrom_frac)`.

#### Signal definition (analysis_frame.fx:308–350)

`achrom_frac` is computed in `MeanChromaPS`:

```hlsl
// analysis_frame.fx:323–330
float in_b = step(0.05, C);          // chromatic gate
achrom_count += 1.0 - in_b;          // counts C < 0.05 pixels
// ...
achrom_frac = achrom_count / float(DS_W * DS_H);   // line 340
```

Stored in `MeanChromaTex.a`, EMA-smoothed at α = `saturate(frametime * 0.005)`
(≈0.083 at 60fps, ≈11-frame time constant). Written to highway slot 202 via
`DebugOverlayPS` (analysis_frame.fx:197–198):

```hlsl
if (xi == HWY_ACHROM_FRAC)
    return float4(tex2Dlod(MeanChromaSamp, float4(0.5, 0.5, 0, 0)).a, 0.0, 0.0, 1.0);
```

No encoding; raw [0,1]. Read via `ReadHWY(HWY_ACHROM_FRAC)`.

#### Existing modulation — R151 (grade.fx:364–368)

`BuildSceneCtx` already modulates `chroma_str_base` by scene chroma level:

```hlsl
float mean_C_scene  = ReadHWY(HWY_MEAN_CHROMA);
ctx.chroma_str_base = CHROMA_STR * 0.04
    * lerp(0.80, 1.20, smoothstep(0.05, 0.35, ctx.zone_log_key))
    * lerp(1.2, 1.0, saturate(mean_C_scene / 0.12));
```

`mean_C_scene` is the Kalman-smoothed median Oklab C of **chromatic pixels only**
(C ≥ 0.05, see analysis_frame.fx:288–350). When median chroma is low the
multiplier is 1.2 (more lift); when elevated it falls to 1.0.

#### Why achrom_frac is not an independent predictor of over-saturation

`achrom_frac` measures the *fraction of neutral pixels in the full scene*, not the
chromatic intensity of the vivid subset. A scene with high achrom_frac can be:

**Case A — uniformly desaturated (foggy, overcast):** achrom_frac ↑, mean_C_scene
also low → R151 already reduces lift. achrom_frac adds no independent information;
both signals agree and the system already handles this correctly.

**Case B — neutral environment + vivid subjects (grey room, character skin/HUD):**
achrom_frac ↑ (driven by neutral walls/floor), but mean_C_scene remains moderate
because it's the *median of chromatic pixels only*. Applying
`lerp(1.0, 0.70, achrom_frac)` globally would reduce chroma lift on the vivid
subject even though those pixels need full treatment. This is the canonical failure
mode: over-squeezing the chromatic minority because the neutral majority pulls
achrom_frac high.

The proposed formula is a global scalar applied at the `chroma_str` level
(grade.fx:540–541), before per-band `LiftChroma`. Every chromatic pixel is
penalised equally regardless of whether it is in a neutral surround or not.
A correction scoped only to neutral-surround pixels would require per-pixel
spatial context — not a scalar highway value.

#### Double-counting risk

Stacking `lerp(1.0, 0.70, achrom_frac)` on top of R151's
`lerp(1.2, 1.0, mean_C_scene/0.12)` creates compounded suppression in
uniformly desaturated scenes where both modulations fire simultaneously:

| Scene         | R151 factor | Achrom factor | Combined |
|---------------|-------------|---------------|----------|
| Near-monochrome (achrom≈0.90, mean_C≈0.03) | 1.20 | 0.73 | 0.88× |
| Overcast (achrom≈0.60, mean_C≈0.07)        | 1.14 | 0.82 | 0.94× |
| Vivid outdoor (achrom≈0.15, mean_C≈0.18)   | 1.00 | 0.90 | 0.90× |

The combined suppression is modest, but the Case B failure (neutral room + vivid
subject) imposes a scene-wide ~20–27% lift penalty on a per-character basis that
has no perceptual justification. This penalty scales with CHROMA_STR and cannot
be dialled out without removing the adaptation entirely.

#### Highway write gap

The `HWY_CHROMA_STR` highway write (grade.fx:655–659) already omits the
mean_C_scene factor present in `BuildSceneCtx`:

```hlsl
if (xi == HWY_CHROMA_STR) {
    float zk = tex2Dlod(ChromaHistory, float4(6.5/8.0, 0.5/4.0, 0, 0)).r;
    float cb = CHROMA_STR * 0.04 * lerp(0.80, 1.20, smoothstep(0.05, 0.35, zk));
    return float4(saturate(cb / 0.10), 0.0, 0.0, 1.0);
}
```

Adding an achrom_frac term to `BuildSceneCtx` would widen this gap further (the
highway slot would under-report the true effective strength by ~0–30%).

#### What a valid implementation would look like (not recommended yet)

If the goal is to reduce lift specifically on near-neutral pixels in a neutral
surround, the correct control point is a **per-pixel** C-weighted gate inside
`LiftChroma`, not a global scalar. For example:

```hlsl
// Inside ApplyChroma, after computing C:
float local_neutral_w = saturate(1.0 - C / 0.08);
float achrom_mod = lerp(1.0, lerp(1.0, 0.70, achrom_frac), local_neutral_w);
chroma_str *= achrom_mod;
```

This zeroes the achrom_frac effect on saturated pixels and applies the full
reduction only to pixels that are themselves nearly neutral — where reducing
chroma lift is harmless by construction. However, this would be a new scope of
work beyond the current candidate definition, and may interact with R22 and R68A.

#### Summary

| Question | Finding |
|----------|---------|
| Is HWY_ACHROM_FRAC on the highway? | Yes — slot 202, raw [0,1] |
| Is it stable? | EMA τ ≈ 11 frames at 60fps; adequate |
| Does it predict over-saturation globally? | No — Case B failure (neutral env + vivid subject) |
| Does it add independent signal over R151? | Partial; redundant in Case A, wrong direction in Case B |
| Can the proposed formula be implemented safely? | No — global application penalises chromatic subjects |
| Should it be re-opened? | Not in global-scalar form; a per-pixel formulation is a separate candidate |

**Candidate closed: REJECT.** The job spec's validation prerequisite ("validate
correlation with subjective over-saturation") cannot be satisfied by code
inspection alone, and the structural analysis demonstrates the signal is
insufficiently correlated with the target outcome in mixed-content scenes.
